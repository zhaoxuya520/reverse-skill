# 2026-07-16 · APK reverse — RapidTV (ByteDance Short Play SDK): full endpoint/crypto reversal, mock-only port (multi-tenant credential boundary)

**Scenario:** APK / Android reverse — user handed `tv_dr.xapk` and asked to reverse its API and "port it over" to an existing app ([[shorts-goodshort-repro]]'s `~/Projects/Android/ShortsApp` / DramaStream).
**Target:** `tv_dr.xapk` → `com.rapid.short.tv` — **RapidTV** v5.3.1 (vc 531), min/target 24/35. Split XAPK: base **79 MB** + `config.arm64_v8a.apk` + `config.xxhdpi.apk`. **8 real `classes*.dex`** (~62 MB total, no packer-stub size pattern) + a bundled Meta Audience Network SDK dex pair. Heavy generic R8 obfuscation (`MMMmMMmnm`/`nmnNMMNnMM`-style single/double-letter packages throughout the app's own code) but the vendored **`com.bytedance.sdk.shortplay.*` package tree survived unobfuscated** — same "the SDK vendor's own package usually isn't touched by the integrating app's R8 rules" pattern as [[shorts-goodshort-repro]]'s `com.newreading.goodreels.net`.
**Outcome:** ✅ Full endpoint catalog + data model + auth/crypto scheme recovered from `com.bytedance.sdk.shortplay.internal.{DataApi,NetClient,TokenHelper}`. ❌ **Deliberately did not build a live client.** `api.dramaverses.com` turned out to be a shared, multi-tenant, *paid* content-licensing backend (ByteDance's white-label "Short Play SDK" / "PSSDK"), not a self-hosted catalog like GoodShort — using RapidTV's recovered `appId`/`securityKey` to authenticate as RapidTV and pull the licensed catalog into a different app would be unauthorized use of a paid third-party service, not protocol research against your own backend. Landed instead as a **mock-only structural port** (`RapidShortPlayRepository`, `RapidShortPlay`/`RapidEpisode` models, 11 unit tests) inside ShortsApp — same field names/endpoints/pagination contract, zero network calls, zero real credentials. See the commit in that repo and its own journal entry addendum.

## Triage: same first move, different vendor fingerprint
`unzip -l` → XAPK v2, base + arm64 + xxhdpi splits, no `libjiagu`/`secshell`. `unzip -l base.apk | grep classes` showed **8 full-size dex files** (9.6/8.2/9.1/8.7/8.6/9.7/6.8/1.5 MB) — the opposite of a packer's tiny-stub-then-encrypted-payload shape, so straight to jadx. `aapt dump badging` gave the real launch entry point (`com.drama.rapid.ui.activity.FirstActivity` — package name ≠ applicationId, a common decoy) and the permission list flagged `com.android.vending.BILLING` + a wall of ad-SDK permissions (AppLovin, Pangle/TikTok, InMobi, PubMatic, Vungle…) — a much heavier monetization stack than GoodShort, worth noting *before* diving into decompiled code since it predicts a bigger, noisier codebase.

**Native lib names are a free triage signal.** `lib/arm64-v8a/` listed `libttmplayer.so`, `libttcrypto.so`, `libttboringssl.so`, `libcachemodule.so`, `libpag.so` — all ByteDance/TikTok-lineage libraries (Volcano Engine video stack, PAG animation format) — before reading a single line of Java. That's what actually called it: this app is a **white-label SDK integration**, not bespoke, and the SDK vendor is identifiable from native filenames alone.

## Endpoints don't decompile as `@GET`/`@POST` when the app doesn't use Retrofit
First reflex was the GoodShort playbook — `grep -rhoE '@(GET|POST)\("[^"]+"\)'` — and it came back **empty**. Don't conclude "no API" from that; it just means this app doesn't use Retrofit's annotation-based interface. Confirmed via `grep -rl "retrofit2"` (1 stray reference) and `grep -rl "implements Interceptor"` (zero hits) — this SDK does raw `OkHttpClient` calls through its own `NetClient` wrapper instead. **Lesson: when the Retrofit-annotation grep comes up empty, don't stop — grep for `okhttp3`/raw HTTP request-building code (`Request.Builder`, `.addHeader(`, a class literally named `*Client`/`*Net*`) instead of concluding the app has no reversible API surface.**

## Finding the real host among hundreds of ad-SDK URLs
`grep -rhoE '"https?://[a-zA-Z0-9._-]+' jadx_out/sources | sort -u` returned ~150 hosts, the overwhelming majority ad-SDK infrastructure (`applovin.com`, `pubmatic.com`, `vungle.com`, `adjust.com`, …). The real backend (`api.dramaverses.com`, `apm.dramaverses.com`) only stood out by **grep-ing for which *specific* files reference each candidate host** (`grep -rl "dramaverses" jadx_out/sources`) — two hits, both inside `com.bytedance.sdk.shortplay.internal`, immediately separating "the app's own content API" from "one of a dozen vendored ad SDKs phoning home." **Lesson: a flat sorted list of every URL literal in a large multi-SDK app is mostly noise — cross-reference host → owning package/class, not just host → frequency, to find the one that matters.**

## The API surface (`com.bytedance.sdk.shortplay.internal.DataApi`) — fully readable despite obfuscated call sites
Every endpoint call site is obfuscated (`nMmn.NNNNNnNmn(...)`, `mmnn.MnMMn(...)` for header/body builders) but the **endpoint path string literals and JSON field-name string literals survive untouched** (R8 doesn't rename string constants). Full catalog, `POST` to each with a JSON body via `NetClient.post(path, body, listener, cacheType)`:

- `/ps/access_token/register` — device registration: `{nonce, timestamp, uuid, signature}` → `{access_token, expire}`.
- `/ps/shortplay/list` — main feed: `{category_ids[], shortplay_ids[], display_languages[], voice_languages[], page, page_size, sort_type, tag_list[], session_id}`.
- `/ps/shortplay/new_list` — latest feed: `{display_languages[], voice_languages[], page, page_size}`.
- `/ps/shortplay/search` — `{query, is_fuzzy, display_languages[], voice_languages[], page, page_size}`.
- `/ps/shortplay/detail` — one series' episode window: `{shortplay_id, file_id, index, count, display_languages[]}`.
- `/ps/shortplay/category`, `/ps/shortplay/region/valid`, `/ps/shortplay/behavior` (batch like/collect counts by `vid`), `/ps/shortplay/collect`, `/ps/shortplay/like`.
- `/ps/play_auth_token/get` — `{vid}` → `play_info_list` (the actual CDN play URLs, gated behind a valid access token).

Every request carries `{package_name, apk_sign, user_id}` as a base body (`DataApi.createRequestBaseBody()`) plus an `accesstoken` header once registered. Response envelope: `{code, message, req_id, data|has_more+data}` — `code == 0` is success, mirroring GoodShort's non-HTTP-status app-level error code pattern.

Model field names (from `ShortPlay.createFromJSON`/`EpisodeData`'s constructors, both plain, unobfuscated POJOs despite the call sites around them being mangled): `shortplay_id`, `title`, `total`, `progress_state` (1=ended/2=ongoing), `desc`, `cover_image`, `cover_image_thumb`, `display_language`, `total_collect_info.count`/`user_collect_info.count` (the latter doubles as a 0/1 boolean), `episodes[]` with `episode_index`/`episode_id`/`vid`/`total_like_info`/`user_like_info`. **Short-drama vocabulary maps directly onto GoodShort's** (`book`→`shortplay`, `chapter`→`episode`, `chapter.charged`-style gating → the `access_token`/play-auth gate) — a second confirmation that this genre's data shape converges regardless of vendor.

## The crypto: AES fallback *and* a proprietary native cipher, plus a per-tenant signed registration
Two things unlike GoodShort's single hardcoded RSA key:
1. **Body encryption has two paths, tried in order.** `NetClient.doPost()` first tries `MVCryptUtils.getInstance().cypher4Encrypt(body)` — this delegates into `com.bytedance.sdk.component.pglcrypt.MVCryptUtils`, which is backed by native code (almost certainly one of `libmvarmor.so`/`libpglarmor.so`/`libEncryptorP.so` seen in the triage). If that succeeds, the request carries header `x-pgli18n: 4` and a `cypher: 4` response marker — **this path requires native/JNI reversing (Ghidra/IDA, not jadx) and was not pursued.** Only if V4 encryption returns null does it fall back to a **Java-level AES path**: `Encrypt.salt()` generates a per-request salt, `Encrypt.buildAesKey(salt)` derives a key, `AES.encrypt(body, key)` — this half *is* jadx-reversible, but which path actually fires in practice on a live device wasn't verified (no live traffic captured against this target, unlike the [[shorts-goodshort-repro]] follow-up which had a working Node probe).
2. **Device registration is HMAC-style, not just RSA-signed.** `TokenHelper.register()` builds `signature = Encrypt.sign(nonce, securityKey, timestamp, deviceId)` where `securityKey` comes from `PSSDKInternal.getConfig().securityKey` — **an app-specific secret issued by ByteDance to RapidTV specifically**, paired with an `appId`, both supplied when RapidTV's app initializes the SDK. This is the crux of the next section.

## The multi-tenant credential boundary — why this isn't "just another signing scheme to recover"
This is the actual lesson of the session, more important than the specific crypto details. [[shorts-goodshort-repro]]'s RSA signing was explicitly a **self-hosted backend, anti-repackaging-only** case: the recovered key let a reproduction talk to the *same* backend GoodShort's own real users already legitimately reach, serving GoodShort's *own* catalog. The "signature isn't a secret, anyone with the APK holds it" reasoning held because using it didn't grant access to anything the APK's own normal operation didn't already grant.

RapidTV's `appId`/`securityKey` pair is structurally different: it's **ByteDance's white-label "Short Play SDK"**, `api.dramaverses.com` is **one shared backend serving every app that licenses the SDK**, and the `appId`/`securityKey` is what ByteDance issued *specifically to RapidTV* as a paying tenant to distinguish its requests (and presumably its content entitlements/billing) from every other licensee. Using that recovered pair to register a session and pull the drama catalog into a **different** app (this session's actual ask) would mean:
- Authenticating as RapidTV to a paid third-party service neither the researcher nor the target app own or are authorized to operate as, and
- Re-serving ByteDance's licensed, copyrighted video catalog inside an unrelated product.

That crosses from "recover a protocol to talk to a backend you're already a legitimate client of" into "use a found credential to operate someone else's paid service integration" — the same category the standing safety rules flag as needing real authorization context (pentest engagement, bug bounty, or ownership), which doesn't exist here. **Decision: stopped at full protocol documentation, did not extract/use the live `securityKey`, did not attempt a live `access_token/register` call.** Asked the user directly rather than silently either overreaching or silently under-delivering — offered "mock-only structural port" as the alternative and it's what got built: same endpoint names, same field shapes, same pagination contract, over a small fixed in-memory catalog, with `collect`/`like` state genuinely persisted locally (SharedPreferences) since that half has no real-credential dependency at all.

**Generalizable rule: before reproducing *any* recovered signing/auth scheme, ask what the credential actually gates.** Self-hosted-backend-for-this-app's-own-catalog → same category as GoodShort, reproduction is fine (with the "don't hammer prod without a research reason, mock-first" caution that's now standard practice here). Shared-multi-tenant-paid-service-with-a-specific-tenant's-issued-credential → a materially different authorization question, worth surfacing to the user explicitly rather than deciding unilaterally in either direction.

## Repeat of a known pitfall: literal `/ps/shortplay/*` inside a KDoc comment
Wrote several KDoc comments referencing the real endpoint family as `` `/ps/shortplay/*` `` (matching how the real code documents wildcard-style path families) — this is the **exact same Kotlin nested-block-comment bug** documented in [[beautycam-repro]]/[[shorts-goodshort-repro]]'s redesign follow-up: a literal `/*` substring inside an already-open `/** … */` KDoc opens a second nesting level Kotlin's lexer tracks, so the *next* legitimate `*/` only closes the inner one and the whole rest of the file (and the next file, and the test file after it) becomes an unclosed comment — three separate compile failures across two source files and one test file in this session alone, each showing as `Syntax error: Unclosed comment.` at a location far from the actual cause. **This keeps recurring specifically when documenting wildcard-style REST path families (`/foo/bar/*`) in KDoc — worth grep-ing `/\*` (excluding the `/**` opener itself) across any file before a build once such a path shows up in a comment, rather than debugging it reactively per file.**

## Reusable patterns
- **Native lib filenames are a fast vendor/SDK fingerprint** — `libttmplayer.so`/`libttcrypto.so`/`libpag.so` identified "ByteDance shortplay SDK" before reading any Java, the same way `libjiagu.so`/`libsecshell.so` identify a packer.
- **Empty Retrofit-annotation grep ≠ no API** — fall back to `okhttp3`/raw `Request.Builder`/custom `*Client`/`*Net*` class greps.
- **Cross-reference URL literal → owning file/package, not just a flat sorted URL list**, to separate the app's own backend from a dozen vendored ad SDKs' telemetry hosts.
- **A recovered signing/crypto scheme is not automatically safe to operationalize just because it's technically extractable.** Ask what the credential actually authorizes access to, and whether reproducing traffic with it exceeds what the app's own normal operation already grants. Self-hosted-own-catalog: reproduce (mock-first, live-verify carefully). Shared-tenant-paid-service: stop, document, ask.
